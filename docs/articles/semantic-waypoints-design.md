# 我给 AI 分身设计了一套记忆系统，没用向量数据库

> 从"百万级检索是伪命题"到"让 LLM 给自己写路标"——一个推荐系统工程师的记忆架构设计手记。

---

## 0. 为什么需要记忆

我有一个 AI 分身，跑在一个叫 OpenClaw 的平台上。它能读我的日志、知道我的决策历史、了解我正在做的项目、认识我的同事。我每天和它聊工作、聊想法，它帮我整理思路、分析信息。

但它有一个根本问题：**没有持久记忆。**

每次对话开始，它都要从头加载上下文。日志越写越多，洞察越积越厚，有一天它不可能把所有东西塞进一个上下文窗口里。

更大的问题是，我希望它不只记得我说过什么，还能看到外面发生了什么——比如 arXiv 上每天几十篇新论文，哪些和我正在做的事有关？能不能主动告诉我？

这就变成了一个推荐系统问题：

```
推荐 = f(你是谁, 你在想什么, 外面发生了什么)
     = f(内部记忆 × 外部记忆)
```

传统推荐系统是"用 1% 的你去匹配 100% 的内容"——一堆行为特征加标签。这个系统是"用 100% 的你去匹配整个世界"——完整的个人记忆、价值观、决策历史。

---

## 1. 第一个直觉：向量检索

做推荐系统的人遇到"大量内容怎么匹配"这个问题，第一反应是上 embedding。

路径很清晰：把论文摘要做 embedding，把用户兴趣做 embedding，算余弦相似度，取 top-K。成熟方案，工业界跑了很多年，开源工具一大把。

我最初也是这么想的。2 月 21 号，我的第一版设计就是双塔向量检索——一套外挂的记忆召回系统。

然后我问了自己一个问题：**这件事，模型足够强以后还需要吗？**

答案是不需要。

向量检索的本质是：模型（或人）不够聪明，没法直接在大量信息里找到想要的，所以需要一个"预筛选层"帮忙缩小范围。但如果模型的上下文窗口足够大、推理能力足够强，它自己就能判断哪些信息相关。

这就是我说的"补偿层"——**补偿模型暂时做不好的事情。**

补偿层有一个致命问题：基座升级后，它变成累赘。你花在向量索引、embedding 模型调优、检索链路上的投资，会随着模型变强而贬值。更糟的是，外挂的匹配层一旦写进系统就很难移除——它会和上下游耦合，变成技术债。

所以我定了一条设计铁律：

> **判断标准：设计任何新功能前先问——"这件事模型足够强后能自己做吗？"**
> - 能 → 不做（补偿层）
> - 不能 → 做（系统能力：持久化、定时拉取、外部数据源、反馈记录）

这不是说向量检索不好——它在很多场景下非常有效。而是说，对于**这个系统**，我希望它的能力天花板等于基座的能力天花板，而不是被任何外挂组件卡住。

## 2. 第二个直觉：让 LLM 直接读

好，不用向量检索。那 LLM 直接读全量数据行不行？

MVP 阶段，完全可行。arXiv cs.IR 每天 5-20 篇论文，一周大约 100 篇，标题加摘要大概 15K tokens。直接全量灌给 agent，让它自己判断哪些和我相关。

但数据会增长。信息源会扩展。当论文库到了几万篇，全量灌入就不现实了。

这时候我走进了一个思维陷阱——觉得问题是"怎么在百万篇中检索"。

后来想通了：**"百万级检索"是伪命题。**

人从来没有"在百万篇中搜索"。人做的是：

1. 订阅特定的信息源（源头筛选）
2. 每天只看增量（定时拉取）
3. 花两秒看标题判断相不相关（快速判断）
4. 用积累的认知做判断（内部知识）

这不是搜索引擎思维，这是**推荐系统思维**：

- 搜索引擎：用户来了 → 在全库里搜 → 返回 top-K
- 推荐系统：内容来了 → 实时决策要不要推给用户

百万级 = 每天几十次小推理 × 时间积累。信息在进入系统的那一刻就该被处理掉。

所以核心不是检索，是**增量处理**。每篇论文进来时，LLM 推理一次，生成一张摘要卡：

```json
{
  "主题标签": ["multi-objective", "reward-shaping"],
  "方法类型": "优化方法",
  "解决的问题": "多目标配平",
  "质量信号": "ICML 2026 accepted",
  "一句话": "不需要手动调 loss 权重的多目标配平方法"
}
```

摘要卡是 LLM 生成的自然语言，不是 embedding。200 tokens 的摘要压缩成 30 tokens 的摘要卡，约 7 倍压缩。可读、可调试、可解释。模型升级后重新生成，就能得到更好的表示——随基座加强而加强。

但问题来了：摘要卡有了，3 万篇的摘要卡池是 90 万 tokens，现在的 1M 上下文刚好能装。但 10 万篇呢？30 万篇呢？

LLM 不可能一次读完所有摘要卡，它需要"先看一部分"。**怎么决定看哪部分？**

如果按语义相关性过滤——那又是外挂匹配层，又是补偿层。

如果按时间或来源过滤——这是"数据导航"，不涉及语义判断，可以接受。

但纯按时间和来源不够精确。有没有一种方式，既不引入外挂匹配层，又能让 LLM 高效地找到相关内容？

## 3. 转折：路标

某天晚上我突然想到一件事。

我自己的记忆系统（OpenClaw 的 `memory/` 目录）已经有一种组织方式了：

```
memory/
├── MEMORY.md              ← 顶层索引（关键词 → 文件映射）
├── insights/              ← 洞察类
├── projects/              ← 项目类
├── decisions.md           ← 决策记录
├── conversations.md       ← 对话记录
└── 2026-03-05.md          ← 日志
```

AI 分身现在怎么用它？读 `MEMORY.md`（顶层索引）→ 找映射 → 加载目标文件 → 逐层深入。

**这就是路标导航。** `MEMORY.md` 是路标，每一行都是 LLM 写给未来的自己的笔记，告诉它"某个话题的详细信息在哪里"。

那如果把这套思路泛化呢？

不再是我手动维护 `MEMORY.md`，而是让 LLM 在处理论文时，自动构建这样一张路标网络。每个节点都是自然语言描述——LLM 写给未来的自己的导航笔记。推荐时，LLM 沿着自己留下的路标导航，而不是系统替它匹配。

**这就是语义路标的核心直觉：让 LLM 在生成摘要卡的同时，构建一张多级语义网。推荐时，LLM 沿着自己写的路标走。**

## 4. 语义路标：怎么运作

### 一张多层语义网

语义路标系统的核心数据结构是一张**多层语义网**。注意不是树，是网。

```
                    ┌─────────────────────────────────────────────┐
  Layer 0           │              顶层路标索引                      │
  (方向层)          │  [推荐系统]  [NLP]  [系统工程]  [因果推断] ...   │
                    └──────┬──────────┬──────────┬────────────────┘
                           │          │          │
  Layer 1  ┌──────────────┐  ┌──────────────┐  ┌──────────────────┐
  (领域层) │ [多目标优化]   │  │ [冷启动]      │  │ [大模型推理]       │
           │ 路标："2026-02 │  │ 路标："生成式  │  │ 路标："..."        │
           │ Pareto 方法   │  │ 召回替代 ID"  │  │                   │
           └───┬───────┬──┘  └───┬──────────┘  └───────────────────┘
               │       │         │
               │       │ ◆═══════╪══════════◆
               │       │ ║  边："方法可用于   ║
               │       │ ║  冷启动场景"     ║
               │       │ ◆═══════╪══════════◆
               │       │         │
  Layer 2      ▼       ▼         ▼
  (论文层)   [A] [B] [C] [F]   [D] [E] [F]
                        ↑                ↑
                        └── 论文 F 同时属于两个节点 ──┘
```

三个关键要素：

**节点**：每个节点是一段自然语言路标。不是标签，不是 embedding，是一句话描述——"2026 年 2 月出现一批 Pareto 方法，核心思路是不手动调权重"。LLM 读这句话就知道这个节点是干嘛的。

**边**：节点之间的关系也是自然语言——"方法可用于冷启动场景"、"共享帕累托数学框架"。不是预定义的关系类型，是 LLM 自由生成的描述。

**多归属**：一篇论文可以同时属于多个节点。一篇关于"用帕累托优化做推荐系统冷启动"的论文，同时挂在"多目标优化"和"冷启动"两个节点下。

节点的数据结构长这样：

```json
{
  "id": "multi-objective",
  "description": "多目标优化：2026-02 出现一批 Pareto 方法，核心思路是不手动调权重",
  "edges": [
    { "to": "cold-start",        "relation": "方法可用于冷启动场景" },
    { "to": "causal-inference",   "relation": "共享帕累托优化的数学框架" },
    { "to": "rec-system",         "relation": "是推荐系统的子领域" }
  ],
  "items": ["论文A", "论文B", "论文C", "论文F"]
}
```

**全部自然语言。** 没有 embedding，没有向量，没有预定义的类型体系。LLM 生成这些描述，未来的 LLM 消费这些描述。

### 导航过程

推荐时，LLM 不做全量检索，而是在这张图上做**推理驱动的游走**：

1. 读用户的兴趣画像（内部记忆）
2. 读 Layer 0 顶层路标索引（约 500 tokens）→ 选择相关方向
3. 进入节点，读邻居和边描述（约 800 tokens）→ 选择下一步
4. 逐层深入，发现跨领域关联（沿着边跳转）
5. 到达论文层，输出推荐列表和推荐理由

10 万篇论文，3-4 层路标导航，LLM 只需要约 3500 tokens 就能到达目标。每一步都是 LLM 推理——系统只负责"打开 LLM 指定的那个节点"，不做任何匹配判断。

```
层级效率分析：

  Layer 0:  1 个根节点        → 索引
  Layer 1:  ~5 个大方向       → 覆盖 100 篇
  Layer 2:  ~25 个细分方向    → 覆盖 500 篇
  Layer 3:  ~125 个节点       → 覆盖 2500 篇
  Layer 4:  ~625 个节点       → 覆盖 12500 篇

  3-4 层覆盖上万篇。每层约 500-800 tokens。
```

### 为什么是网不是树

这是一个有意识的设计选择。

| | 树 | 网 |
|---|---|---|
| 到达路径 | 唯一（从根到叶） | **多条**（从不同节点都能到达同一篇论文） |
| 发现关联 | 只能发现"同一分支下"的关联 | **能发现跨领域关联**（沿着边跳转） |
| 导航决策 | "进入哪个子节点" | **"走哪条边"** — 更丰富的推理空间 |
| 多归属 | 论文只能放一个位置 | **论文被多个节点引用**，天然解决 |
| 知识表达 | 层级分类（强制互斥） | **关系网络**（反映知识的真实结构） |

知识本来就不是树状的。一篇论文可以同时关于推荐系统、因果推断和强化学习。强制把它放在一个分支下，就丢失了跨领域的关联。网状结构天然支持多归属和关联发现——这更接近知识的真实形态。

## 5. 为什么这不是另一种 RAG

你可能在想：这和 RAG（Retrieval-Augmented Generation）有什么区别？或者和 GraphRAG、Mem0 这些方案有什么不同？

关键差异在一个问题上：**谁在做匹配决策？**

| | 向量检索 (RAG) | GraphRAG | Mem0 | 语义路标 |
|---|---|---|---|---|
| 谁生成表示 | 外挂 embedding 模型 | LLM 提取实体 | LLM 提取 | **LLM 生成路标** |
| 谁做匹配 | **系统**（余弦相似度） | **系统**（向量匹配） | **系统**（向量 + 图检索） | **LLM**（推理导航） |
| 基座升级时 | 检索仍走向量 | 检索仍走向量 | 向量检索不变 | **生成 + 消费都加强** |

所有这些方案，在匹配环节都保留了至少一层**系统级组件**——embedding、向量距离计算、图查询引擎。不管生成索引的 LLM 有多强，最终"这篇论文和用户相关吗"这个判断，是系统用数学公式或预定义规则做的。

语义路标的不同在于：**匹配判断也交给 LLM。** LLM 读路标描述，推理"这个方向和用户兴趣相关吗"，然后决定走向哪里。没有中间的系统匹配层。

这意味着：系统的能力天花板 = 基座的能力天花板。GPT-5 出来了，路标导航自动变好——生成的路标描述更精准，导航推理更准确。不需要换 embedding 模型，不需要重新调向量索引，不需要更新图查询逻辑。

我把这叫做 **LLM-native architecture**：索引构建、结构演化、导航检索全部交由 LLM 推理完成，不依赖任何外挂匹配机制。

## 6. 结构应该是活的

到目前为止，语义路标和一些已有方案（比如 RAPTOR、SHIMI）看起来有相似之处——都是多层索引结构，都用 LLM 生成摘要/描述。

真正的分野在"自生长"。

RAPTOR 是一次性自底向上构建：拿到一批文档，递归聚类，生成摘要树。如果新增了 1000 篇论文，需要重建整棵树。GraphRAG 类似——一次性提取实体、社区检测、生成社区摘要。新数据来了，重建。

语义路标的设计理念不同：**知识结构应该是活的，随数据持续演化，而非一次定型。**

### 冷启动从零开始

第 0 天，语义网是空的。第 1 篇论文入库，LLM 创建第 1 个节点。前 1-2 周，大部分论文挂在同一个节点下，归类粗糙——但这没关系，论文少的时候粗糙的代价极低。

### 日常入库：轻量 O(1)

新论文进来时，LLM 只做两件事：
1. 生成摘要卡
2. 归到现有节点中最匹配的那个

不改网络结构，不迁移论文。即使归类不完美，暂挂最近似节点，等后续调整。

### 定期重整：全局审视

当某个节点积累了太多论文（比如超过 20 篇），或者近期"低置信度归类"比例过高，触发重整。重整时 LLM 全局审视：

- 一个节点下的论文其实讲了 3 件不同的事 → **分裂**成 3 个子节点
- 两个节点的内容趋同 → **合并**
- 发现两个看似不相关的方向其实共享某个数学框架 → **建边**
- 路标描述过时了 → 更新

分裂后的子节点自然形成新的层级——层级不是预先规定的，是分裂的自然产物。热门方向论文多，分裂多，自然长得更深；冷门方向保持浅层。

### 入库时的"微感知"

这里有个巧妙的设计。LLM 归类论文时，顺便输出一个 side signal：

```json
{
  "归类到": "推荐系统",
  "归类置信度": "低",
  "感知信号": "这篇和节点下其他论文主题不太一样"
}
```

成本几乎为零——LLM 归类时多输出几个 token。这些微弱信号积累起来，就是触发重整的依据。

和人的类比：**每天随手归档到"大概是这个方向"的文件夹，定期花半天重新整理。**

### 完整的生命周期

```
第 0 天：冷启动
  → LLM 创建 [推荐系统] 节点 → 论文 A 挂上

前 1-2 周（冷启动期）
  → 大部分论文挂在同一个节点下

积累到 ~20 篇
  → 置信度偏低，触发第一次重整

重整过程
  → LLM 读摘要卡 → "这些论文讲 3 件不同的事"
  → 分裂节点 + 建跨领域边 + 迁移论文

分裂后的日常
  → 新论文入库，归类更精准，置信度回升
  → 继续积累 → 下一次重整 → 循环
```

这不是工程优化。这是**根本理念不同**——知识结构不应该是一次性构建的静态产物，它应该像一棵树一样生长。

## 7. 双记忆：内部驱动外部

到这里，语义路标解决了"怎么在大量论文中导航"的问题。但记忆系统不只有外部信息。

回到核心公式：

```
推荐 = f(内部记忆 × 外部记忆)
```

内部记忆（Inner Memory）是关于"我"的一切——日志、决策、洞察、人物关系、项目上下文。外部记忆（Outer Knowledge）是关于"世界"的信息——论文、技术博客、行业动态。

**它们不是对称的两套系统。内部是需求侧，外部是供给侧。内部驱动外部。**

```
┌─────────────────────────────────────────────────┐
│              语义路标引擎（共享）                   │
│  归类 · 导航 · 重整 · 自生长 — 同一套机制          │
└──────────┬──────────────────────┬────────────────┘
           │                      │
   ┌───────▼────────┐    ┌───────▼────────┐
   │  内部记忆 Store  │    │  外部记忆 Store  │
   │  日志/决策/洞察   │    │  论文/资料        │
   └───────┬─────────┘    └───────▲─────────┘
           │                      │
           │  ① 驱动检索           │
           │  生成精准 query ──────→│
           │                      │
           │  ② 驱动淘汰           │
           │  兴趣变化 → 清理 ────→│
           └──────────────────────┘
```

### 同一套引擎，隔离数据

内部记忆和外部记忆用**同一套语义路标机制**——归类、重整、导航的逻辑完全共享。但数据严格隔离：各自独立的路标图（graph.json）、各自独立的卡片库（cards/）、各自独立的写入管道。

为什么必须隔离？因为**身份边界不能污染**。内部记忆描述的是"你是谁"，外部记忆描述的是"世界有什么"。混在一起，LLM 会分不清哪些是你的观点、哪些是论文的结论。

实际的数据路径：
- 论文路标图：`~/.openclaw/personal-rec/knowledge/graph.json`（70 节点，2925+ 论文）
- 记忆路标图：`~/.openclaw/memory-index/knowledge/graph.json`（19 节点，219+ 记忆实体）

### 内部记忆怎么建索引

内部记忆当前是散装 Markdown 文件。索引构建过程：

1. **解析**：按 Markdown 标题（`##`、`###`）拆分 section，每个 section 生成一张卡片
2. **日期提取**：三层优先级——section 标题中的日期 > 父标题继承 > 文件名中的日期
3. **人物识别**：从身份速查表加载人物词典（企微 ID + 中文姓名），正则匹配
4. **冷启动归类**：根据源文件路径自动归类到 7 个 seed 节点（decisions、insights、daily-log、projects 等）
5. **LLM 重整**：积累足够多后，LLM 审视并分裂

实际重整效果：

```
重整前：7 个 seed 节点
  daily-log(73)  archive(54)  projects(35)  decisions(19)  insights(17)

重整后：19 个节点（seed + LLM 分裂子节点）
  daily-log 分裂为 →
    team-collaboration(25)  recommendation-system(15)  scaling-up-tech(15)
    mcp-architecture(9)  experiment-data(5)  openclaw-dev(3)
  archive 分裂为 →
    memory-system-architecture(18)  technical-implementation(15)
    ai-project-management(6)  creator-tools(6)
```

从 7 个粗糙的目录分类，自动演化成 19 个有语义含义的方向。节点名不是学术概念（那是外部论文的风格），而是日常用语——`team-collaboration`、`scaling-up-tech`。因为这是关于"我"的记忆，用我的语言来组织。

### 内部驱动外部：检索

内部记忆重整后，系统从 19 个方向中识别出 15 个**活跃方向**（最近 14 天有新增卡片的节点）。这 15 个活跃方向就是"我当前在关注什么"。

然后 LLM 读每个活跃方向的描述和样例，生成学术检索 query：

```
活跃方向："recommendation-system" — 推荐系统的多目标优化和冷启动策略
近期样例：["帕累托方法配平", "冷启动生成式召回"]

→ LLM 生成 query:
  ["multi-objective optimization recommendation system",
   "cold-start generative retrieval"]
```

这些 query 送到 Semantic Scholar API 搜索论文，搜回来的论文去重后接入已有的入库管道——生成摘要卡，归类到外部论文的路标图。

**整个循环**：内部记忆变化 → 关注方向更新 → 检索 query 更新 → 外部论文库更新。内部驱动外部，持续运转。

### 内部驱动外部：淘汰

检索只解决"扩充"，还需要"收缩"。当兴趣方向迁移，之前关注的论文不再相关。

两种淘汰模式：

**主动淘汰**：内部记忆重整后，发现某个方向的卡片数减少超过 50% → 该方向下的论文被标记为"过期"。

**被动淘汰**：论文库规模超限（默认 10000）→ 按综合得分排序（与活跃方向的相关度 × 0.5 + 导航命中次数 × 0.3 + 入库时间衰减 × 0.2）→ 清理排名最低的。

淘汰不是"删除"，是"降低可达性"——先从导航路径移除，再归档卡片，物理删除默认不做。可逆、渐进、透明。

## 8. 卡片上该放什么：一次痛苦的精简

写到这里，似乎一切都很顺畅。但实际做的时候，我在一个看似不起眼的问题上纠结了很久：**卡片上到底该放哪些字段？**

最初的设计，每张记忆卡片长这样：

```typescript
interface ContentCard {
  id: string;
  title: string;
  oneLiner: string;
  tags: string[];           // 主题标签
  projects: string[];       // 关联项目
  locations: string[];      // 场景（周会、1v1等）
  people: string[];         // 相关人物
  qualitySignal: string;    // 质量评分
  type: string;             // 类型分类
  relatedEntities: string[];// 关联实体
  // ...
}
```

字段很多，看起来很"完整"。但验收的时候发现两类问题：

**问题 1：规则匹配不可靠。** 项目关联是通过字符串匹配做的——`text.includes("沟通白盒")`。但一段内容末尾引用了另一个项目的链接，也会被匹配上。归属 vs 提及，系统分不清。

**问题 2：中英文重复。** 项目词典从文件名提取了英文 slug（`communication-whitebox`），从一级标题提取了中文名（`沟通白盒`）。两个字符串被视为两个不同的项目。

直觉反应是修 bug：加 canonical ID 做去重，加 owner project 从路径推导归属。

但回头看设计原则——**谁在做决策？** 这些修复方案的本质是让系统用更精细的规则来做"这段内容和哪个项目相关"的判断。但这是语义理解任务，应该由 LLM 做。

那让 LLM 做？对每张卡片调一次 LLM 判断项目归属？几百张卡片，成本太高。

最终我做了一个更根本的决定：**砍掉这些字段。**

```
砍掉的：tags、projects、locations、relatedEntities、qualitySignal、type
保留的：id、title、oneLiner、people、source、date
```

理由：关联关系不应该在卡片上冗余存储——**交给图拓扑表达**。一张卡片属于哪个项目？看它在哪个节点下。和什么主题相关？看它的节点路标描述。跨主题关联？看节点的边。

**唯一保留在卡片上的实体属性是 `people`**——因为人名是稳定的标识符，词典匹配足够准确；而且"某人相关的所有内容"这种查询模式需要从卡片维度检索，节点拓扑不适合表达（一个人跨很多节点）。

这次精简教会我一个原则：**卡片上每个字段都必须回答"它为什么必须存在"。不能回答就不该留。确定性字段（事实性元数据 + LLM 语义摘要）放卡片上，所有关联关系交给图拓扑。**

## 9. 和业界方案的比较

做到这里，我回头系统性地梳理了一遍业界的相关工作，看看哪些人在做类似的事，我们的设计选择和他们有什么不同。

### 能力矩阵

| 能力 | SHIMI | Mnemis | RAPTOR | MemWalker | GraphRAG | Mem0 | **语义路标** |
|------|-------|--------|--------|-----------|----------|------|-------------|
| 层次化索引 | ✅ | ✅ | ✅ | ✅ | — | — | ✅ |
| LLM 生成索引 | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| LLM 导航（非向量） | — | 部分 | — | ✅ | — | — | **✅** |
| 图（非树）拓扑 | — | — | — | — | ✅ | ✅ | **✅** |
| 自然语言边 | — | — | — | — | 预定义 | — | **✅** |
| 增量自生长 | 部分 | 不详 | ❌ | ❌ | ❌ | — | **✅** |
| 全链路零外挂 | — | — | — | — | — | — | **✅** |

没有人做过的组合：**LLM 生成自然语言路标 → 构成多层语义网 → LLM 沿路标推理导航 → LLM 自主重整 → 全链路零外挂。**

### 最近缘的三个工作

**SHIMI**——最相似的结构设计。也做语义层次索引，但它是树结构 + 系统检索，重心在去中心化多 agent 同步。我们是网状结构 + 全链路 LLM。

**MemWalker**——最相似的导航直觉。LLM 也是主动在结构上"走"的。但 MemWalker 做的是单文档上下文压缩（把长文档压成导航树），不做知识组织，没有自生长、没有网状结构、没有边关系。

**RAPTOR**——最相似的构建方法。也用 LLM 生成摘要节点。但 RAPTOR 是静态的一次性构建 + 向量检索。我们是动态自生长 + LLM 导航。

### 为什么自然语言边很重要

GraphRAG 也用图结构，但它的边是预定义的实体关系类型。你得提前告诉系统有哪些关系类型（"属于"、"使用"、"创建"等）。

问题是：跨领域的关联往往不在任何预定义类型里。一篇论文用帕累托优化做推荐系统的冷启动——"多目标优化"和"冷启动"之间的关系是"共享帕累托数学框架"。这个关系不在任何标准的关系类型库里。

语义路标的边由 LLM 自由生成自然语言描述。它可以写出"方法可用于冷启动场景"、"共享帕累托优化的数学框架"、"是...的子领域"——任意类型的关系，任意粒度的描述。

## 10. 系统能力 vs 补偿层

最后我想谈谈设计哲学。

整个系统的设计可以用一个问题来审视每一个组件：**这件事，模型足够强后能自己做吗？**

**模型做不了的（系统能力）**：
- 持久化——模型没有跨对话的记忆，需要系统存储
- 定时拉取——模型不会自己去拉 RSS，需要系统调度
- 外部数据源——模型不知道 arXiv 有新论文，需要系统告诉它
- 反馈记录——模型不记得你上次忽略了什么，需要系统记录
- 淘汰调度——模型不知道什么时候该清理，需要系统触发

**模型能做的（基座能力）**：
- 判断一篇论文是否相关
- 生成摘要
- 归类到合适的节点
- 写路标描述
- 导航时选择方向
- 发现跨领域关联
- 重整时决定分裂还是合并

**补偿层（模型暂时做不好，但未来能做的）**：
- 向量检索——现在模型上下文不够长，需要向量预筛选；未来上下文增长后可能不需要
- embedding 匹配——现在模型推理太慢，需要快速预过滤；未来推理成本下降后可能不需要
- 规则路由——现在模型判断不够稳定，需要规则兜底；未来推理更稳定后可能不需要

语义路标的设计选择是：**只投资系统能力，拒绝补偿层。** 系统做系统擅长的事（持久化、调度、数据导航），模型做模型擅长的事（理解、推理、判断、生成）。两者之间不插入任何中间的匹配层。

这个选择有代价。现在的延迟比向量检索高（4 层路标导航约 4-8 秒 vs 向量检索毫秒级）。但我的判断是：**上下文窗口持续增长 + 推理成本持续下降，效率差距会缩小，而架构差距不会——外挂一旦写进系统就很难移除。**

## 11. 实际运行的样子

说了这么多架构和哲学，实际跑起来是什么样？

**论文入库**（每天自动运行）：
1. 定时拉取 arXiv RSS
2. 每篇论文生成摘要卡（~30 tokens，一次 LLM 调用）
3. 归类到路标图中最匹配的节点（一次 LLM 调用）
4. 如果内部记忆有更新，从活跃方向生成检索 query，搜索 Semantic Scholar，新论文入库
5. 当前：70 个节点，2925+ 篇论文

**记忆索引**（增量更新）：
1. 检测 memory/ 目录下的文件变更
2. 新增 section 生成卡片，归类到路标图
3. 当前：19 个节点，219+ 记忆实体

**推荐**（按需 + 定时）：
1. 读内部记忆的 15 个活跃方向
2. 在外部论文路标图上导航
3. LLM 生成推荐列表和推荐理由
4. 每条推荐带理由——"因为你在关注多目标优化，这篇论文提出了不需要调权重的帕累托方法"

整个系统的代码在一个 TypeScript 项目里，零外部数据库依赖，数据全部是 JSON 文件存在本地，跑在 OpenClaw 平台上。

## 12. 回顾

从 2 月 21 号的"双塔向量检索"到现在的"语义路标 + 双记忆架构"，思路转了三次弯：

| 时间 | 方向 | 放弃原因 |
|------|------|---------|
| 2/21 | 双塔向量检索 | 补偿层——模型长上下文变强后被淘汰 |
| 2/28 上午 | 轻量模型做记忆路由 | 也是补偿层——模型变强后自己能路由 |
| 2/28 下午 | 个人分身推荐系统 | 对了，但缺少结构化的记忆组织机制 |
| **3/5** | **语义路标记忆系统** | 当前方案 |

核心洞察就两个：

1. **百万级检索是伪命题**——信息在进入系统的那一刻就该被处理掉，这是推荐系统思维，不是搜索引擎思维。

2. **让 LLM 给自己写路标**——生成者和消费者都是 LLM，中间不插任何系统匹配层。结构由 LLM 自生长，系统只做数据导航。

这个系统现在还很早期。路标导航的延迟、重整的质量、推荐的精准度，都有大量可以优化的空间。但架构层面，我对设计选择的信心越来越强：**把赌注押在基座能力的持续提升上，而不是在当前基座的弱点上修补丁。**

模型会越来越强。系统能力不会自己长出来。

所以，投资系统能力，拒绝补偿层。
