## 一、问题定义

现有 RAG + 向量检索在三个维度失败：

1. **抽象能力缺失** — embedding 是扁平的，无法表达概念层次（"机器学习"包含"强化学习"这层关系在向量空间里是隐式的）
2. **集中式索引** — 与去中心化信任模型冲突（多 Agent 场景下谁拥有索引？）
3. **不可解释** — 向量距离是黑盒，无法回答"为什么检索到这条"

## 二、核心方法

### 2.1 语义树 $T = (V, E)$

SHIMI 将知识组织为**动态语义树**，4 个关键参数：

| 参数 | 含义 | 作用 |
|------|------|------|
| $T$ | 分支因子（每个父节点最大子节点数） | 控制树的宽度 |
| $L$ | 最大抽象层级数 | 控制树的深度 |
| $\delta \in [0,1]$ | 相似度阈值 | 放置/匹配时的门槛 |
| $\gamma \in [0,1]$ | 语义压缩率 | 摘要时的词汇缩减比例 |

每个节点包含：
- **语义摘要** $s(v)$ — 该子树涵盖内容的自然语言概述
- **子节点列表** $C(v)$
- **实体集合** $\mathcal{E}_v$ — 叶子节点挂载的具体内容
- **父指针** $p(v)$

### 2.2 算法 1：AddEntity（实体插入）

```
输入：新实体 e
1. MatchToBucket(e)     → 在根节点层找到候选桶
2. DescendTree(R, e)    → 沿树语义下降，找到最匹配的叶子
3. 附加到叶子节点
4. if |C(v)| > T:       → 子节点数超过分支因子
     MergeConcepts(v_i, v_j)  → 合并两个最相似的子节点为新父节点
```

**关键机制**：当子节点过多时，不是分裂，是**合并**——找两个最语义相近的节点合并到一个新抽象父节点下。这是自底向上的层次生成方式。

### 2.3 算法 2：RetrieveEntities（语义检索）

```
输入：查询 q
1. 从根节点开始
2. 对每个子节点 v，计算 sim(q, s(v))
3. 仅展开 sim ≥ δ 的分支       → 语义剪枝
4. 递归直到叶子
5. 返回匹配的实体集合
```

自顶向下遍历，核心是**语义剪枝**——不相关的子树整棵跳过。

### 2.4 算法 3：SHIMI_PartialSync（去中心化同步）

这是 SHIMI 真正的重心所在——多 Agent 之间的记忆同步：

```
Agent_i 和 Agent_j 各自维护本地语义树 T_i, T_j

Step 1: 差异检测
  比较 Merkle 根哈希 H_i, H_j
  不同 → 递归识别最小差异子树 T_d

Step 2: 概率过滤
  Agent_i 生成 T_d 的 Bloom Filter B_i 发送给 Agent_j
  Agent_j 用 B_i 推断自己缺哪些节点（避免全量扫描）

Step 3: 冲突解决
  合并函数 μ(v_i, v_j) 满足 CRDT 属性（交换律、幂等律、结合律）
  冲突时优先选择：抽象深度更深 or 使用频率更高的摘要
```

**同步只传差异**——Merkle-DAG 定位差异、Bloom Filter 减少扫描、CRDT 保证最终一致性。

## 三、实验结果

### 3.1 检索准确度（20 个语义非平凡查询）

| 方法 | Top-1 准确率 | 平均 Precision@3 | 可解释性 (1-5) |
|------|-------------|-----------------|---------------|
| **SHIMI** | **90%** | **92.5%** | **4.7** |
| RAG 基线 | 65% | 68.0% | 2.1 |

### 3.2 遍历效率

随树深度增加（2→5），SHIMI 的每次查询访问节点数呈**次线性增长**（有效剪枝），RAG 扁平扫描接近线性增长。

### 3.3 同步带宽

| 节点数 | SHIMI 同步 (KB) | 全量同步 (KB) | 节省 |
|--------|----------------|--------------|------|
| 3 | 118 | 1320 | **91%** |
| 6 | 248 | 2650 | **90.6%** |

### 3.4 可扩展性

实体增长到 2000 个时，SHIMI 查询延迟保持**相对平坦**，RAG 线性增长。

### 3.5 局限性（论文自述）

1. **严格树状结构**——难以处理多重分类或多图表示
2. **语义推理依赖 LM**——缺乏形式化保证
3. **模拟环境评估**——未考虑真实网络延迟或对抗性输入

## 四、与语义路标的深度对比

| 维度 | SHIMI | 语义路标 |
|------|-------|---------|
| **节点内容** | 结构化语义摘要 $s(v)$ + 参数化元数据 | **全自然语言**路标（LLM 写给自己的笔记） |
| **谁做匹配** | `sim(q, s(v))` — 系统计算相似度 | **纯 LLM 推理**读路标后自主决定 |
| **拓扑** | 严格树 $T=(V,E)$ | **网**（多归属 + 自然语言边） |
| **结构演化** | 子节点超过分支因子 $T$ 时合并 | LLM 自主分裂/合并/建边，两阶段触发 |
| **多归属** | ❌ 树结构天然不支持 | ✅ 论文可被多个节点引用 |
| **跨领域发现** | ❌ 只能发现同分支下的关联 | ✅ 沿自然语言边跳转到关联领域 |
| **核心重心** | 去中心化同步协议（多 Agent） | 单用户全链路 LLM 推理能力 |
| **同步机制** | Merkle-DAG + Bloom + CRDT | 不涉及（单用户场景） |
| **可演进性** | 受 $T, L, \delta, \gamma$ 参数约束 | 随基座增强自然增强 |
| **评估规模** | 2000 实体 | 设计面向 10 万+ |

## 五、对我们的借鉴

### ✅ 可直接吸收的设计

**1. 语义剪枝的思路（Algorithm 2）**

SHIMI 的检索核心是 `sim(q, s(v)) ≥ δ` 剪枝——不相关子树整棵跳过。我们的语义路标导航本质上在做同样的事，但用的是 **LLM 推理**而非系统计算相似度。

→ **借鉴**：导航 prompt 中可以明确引导 LLM 做"剪枝式"思考——先扫描所有子节点路标描述，快速排除不相关的，再深入相关的。这比让 LLM 逐个判断效率更高。

**2. 实体插入时的"合并"机制**

SHIMI 子节点超过分支因子时，找两个最相似的合并成新抽象父节点。这是**自底向上**的层次涌现方式。

→ **借鉴**：我们的重整机制目前只设计了"分裂过大节点"，没有显式设计"合并过于相似的节点"。SHIMI 的合并思路可以补充——重整时 LLM 除了判断"该不该拆"，也可以判断"这两个节点是不是太像了该合并"。加到重整 prompt 的检查项中。

**3. 复杂度分析框架**

SHIMI 给出了清晰的复杂度公式：插入时 API 调用 ≈ $R + 0.5 \cdot A \cdot T \cdot d(d+1)$，$n=10^6$ 时约 35 次调用。

→ **借鉴**：我们在 technical-design.md 中做了类似的层级-token 估算（4 层 ~3500 tokens），但缺少对"入库"操作的复杂度分析。可以补充：入库时 LLM 读多少路标 × 几次调用。

### ⚠️ 需要警惕的设计选择

**1. 相似度阈值 $\delta$ — 硬编码门槛**

SHIMI 用固定阈值 $\delta$ 做剪枝决策。这本质上是**系统在做决策**——阈值是人定的，不随内容语义变化。

→ **我们的选择**：不设阈值，由 LLM 推理判断"这个方向值不值得探索"。代价是不可预测的行为，但收益是随基座增强而增强。这个差异是我们的核心定位之一。

**2. 严格树结构 — SHIMI 自己承认的局限**

论文 Limitations 明确说"当前假设严格树结构，难以处理多重分类"。这恰好是我们用网状结构解决的问题。

→ **注意**：SHIMI 未来工作提到"将语义模型推广至非树状图和循环本体"——如果他们做了，和我们的差异会缩小。需要持续跟踪。

**3. 分支因子 $T$ 和层级数 $L$ — 预定义参数**

SHIMI 需要预先设定这些参数来控制树的形状。这和我们"不预定义层数，由分裂机制自然生长"的设计相悖。

→ **定位价值**：我们的"自适应结构"比 SHIMI 的"参数化结构"更激进地信任基座能力。

### 💡 启发的新思考

**1. Merkle-DAG 做变更检测 — 未来多用户场景可能有用**

虽然我们现在是单用户，但 OpenClaw 的 workspace 本身是 Git 同步的。如果未来多个 Agent/分身需要同步语义网的修改，Merkle 哈希做差异检测是轻量级的好方案。不需要现在做，但记住这个方向。

**2. 论文的评估指标可以复用**

SHIMI 用了 Top-1 准确率、Precision@3、可解释性评分。我们的 checklist 目前缺少定量评估指标，可以参考这个框架设计 MVP 的评测。

**3. "语义保真度"这个概念值得关注**

SHIMI 强调检索结果要保持语义保真度（不只是表面匹配，要真正理解意图）。我们的导航靠 LLM 推理天然有这个优势，但在评估时应该把"语义保真度"作为一个显式的评估维度。

## 六、一句话定位

> SHIMI 证明了"语义层次 + 自顶向下遍历"比扁平向量检索显著更好（Top-1: 90% vs 65%），但它的匹配仍由系统完成、拓扑限制为树。语义路标在同样的层次遍历思路上更进一步：**LLM 全程做决策 + 网状多归属 + 自生长结构**。
