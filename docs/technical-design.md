## 十、核心课题：百万级数据下的推荐架构

### 问题

MVP 阶段，arXiv cs.IR 每天 5-20 篇，一周 ~100 篇，标题+摘要 ~15K tokens，全量灌给 agent 完全可行。

但当 feed 源扩展、时间积累，数据量到数千甚至百万级，全量灌入不现实了。怎么办？

传统路径是 embedding → 向量检索 → top-K → 排序模型。但这违反设计原则（外挂检索）。

### 关键洞察：这不是一个检索问题

**"百万级检索"是伪命题。** 真正的问题不是"怎么在 100 万篇中搜"，而是"怎么让每篇信息进来的时候就被处理掉"。

人也从来没有"在百万篇中搜索"。人做的是：
1. **订阅特定源**（源头筛选）→ 系统能力
2. **每天只看增量**（定时拉取，按日存储）→ 系统能力
3. **2 秒看标题判断**（LLM 读标题+摘要）→ 基座能力
4. **积累认知做判断**（Inner Memory）→ 基座能力

这是**推荐系统思维**，不是搜索引擎思维：
- 搜索引擎：用户来了 → 在全库里搜 → 返回 top-K（外挂检索）
- 推荐系统：内容来了 → 实时决策要不要推给用户（增量处理）

**百万级 = 每天几十次小推理 × 时间积累。** 信息在进入系统的那一刻就被处理了。

### 单用户方案

```
每天 20 篇新论文入库
  → LLM 逐篇读（标题+摘要，几千 tokens，轻松）
  → 结合 Inner Memory 判断："这篇对你有价值吗？"
  → 有价值的进入"待推荐池"
  → 对话时 / 每日 briefing 时，从池中挑选推荐
```

- LLM 每次只处理几十篇（几千 tokens）
- 判断依据是 Inner Memory（基座能力）
- 有价值的标记为候选，没价值的过期清理
- 推荐时只面对"候选池"（几十 ~ 几百篇），不是百万级

### 多用户扩散读问题

单用户是 fanout-on-write：一篇论文入库 → 1 次 LLM 推理。完美。
多用户时：一篇论文入库 → N 个用户各做 1 次推理 = **O(N) 扩散读**。

这和推荐系统里经典的 fanout 问题完全同构：

| | 推荐系统 | 分身推荐 |
|---|---|---|
| fanout-on-write | 内容发布时推送到每个粉丝 feed | 论文入库时对每个用户做推理 |
| fanout-on-read | 用户刷新时实时从关注列表拉 | 用户需要推荐时实时推理 |

### 思路演进：从标签匹配到全链路 LLM 推理

#### 第一版想法：共性推理 + 标签交集（已放弃）

最初的思路是：LLM 入库时生成摘要卡（含主题标签），推荐时用标签交集做粗筛，再用 LLM 精排。

**问题**：标签交集是**系统在做匹配**——LLM 生成了标签，但消费标签的是系统（集合操作）。这本质上还是一层外挂：模型越强，这层越多余。而且标签匹配是粗糙的字符串级别，丢失了语义关联。

#### 追问：匹配能不能也是 LLM 推理？

去掉标签交集后，LLM 直接读摘要卡池 + 用户画像 → 推理输出推荐。完全可行：摘要卡 ~30 tokens/篇，1M 上下文能装 3 万篇。

**但新问题**：当数据量超过上下文窗口，LLM 需要"先看一部分"。怎么决定看哪部分？

- 按时间/源过滤 → 这是"数据导航"，合理
- 按语义相关性过滤 → 这又是"系统做匹配"，外挂

#### 最终方案：语义路标 + LLM 导航

核心洞察：**让 LLM 在生成摘要卡的同时，构建一棵多级知识地图。每个节点都是自然语言"路标"——LLM 写给未来的自己的笔记。推荐时，LLM 沿着自己留下的路标导航，而不是系统替它匹配。**

### 转化方案：共性推理 × 语义路标 × LLM 导航

#### 第一层：共性推理 — O(1)，全局共享

论文入库时，LLM 推理一次，生成**摘要卡**：

```json
{
  "主题标签": ["multi-objective", "reward-shaping"],
  "方法类型": "优化方法",
  "解决的问题": "多目标配平",
  "质量信号": "ICML 2026 accepted",
  "一句话": "不需要手动调 loss 权重的多目标配平方法"
}
```

- 摘要卡是 **LLM 生成的自然语言**，不是 embedding（可读、可调试、可解释）
- 对所有用户共享，只推理 1 次
- 模型升级后重新生成即可获得更好的表示——**随基座加强而加强**
- **信息压缩比**：原始摘要 ~200 tokens → 摘要卡 ~30 tokens，约 7 倍压缩

#### 第二层：语义路标 — LLM 构建多层语义网

LLM 处理增量论文时，不只生成摘要卡，还维护一张**多层语义网**（不是树，是网）：

```
多层语义网（LLM 生成 + LLM 导航）

知识不是树状的，知识是网状的——概念之间有多种关系、多个角度的关联。

                    ┌─────────────────────────────────────────────┐
  Layer 0           │              顶层路标索引                      │
  (方向层)          │  [推荐系统]  [NLP]  [系统工程]  [因果推断] ...   │
                    └──────┬──────────┬──────────┬────────────────┘
                           │          │          │
            ┌──────────────┘          │          └──────────────┐
            ▼                         ▼                         ▼
  Layer 1  ┌──────────────┐  ┌──────────────┐  ┌──────────────────┐
  (领域层) │ [多目标优化]   │  │ [冷启动]      │  │ [大模型推理]       │
           │ 路标："2026-02 │  │ 路标："生成式  │  │ 路标："..."        │
           │ Pareto 方法   │  │ 召回替代 ID"  │  │                   │
           └───┬───────┬──┘  └───┬──────────┘  └───────────────────┘
               │       │         │
               │       │ ◆═══════╪══════════◆
               │       │ ║  边："方法可用于   ║
               │       │ ║  冷启动场景"     ║
               │       │ ◆═══════╪══════════◆
               │       │         │
  Layer 2      ▼       ▼         ▼
  (论文层)   [A] [B] [C] [F]   [D] [E] [F]
                        ↑                ↑
                        └── 论文 F 同时属于两个节点 ──┘
```

**节点**：每个节点是一段自然语言路标（LLM 写给未来的自己的笔记）
**边**：节点之间的关系也是自然语言（"方法可用于"、"共享数学框架"、"是...的子领域"）
**论文**：可以被多个节点引用（网状天然支持多归属）

节点数据结构示例：

```json
{
  "id": "multi-objective",
  "description": "多目标优化：2026-02 出现一批 Pareto 方法，核心思路是不手动调权重",
  "edges": [
    { "to": "cold-start",        "relation": "方法可用于冷启动场景" },
    { "to": "causal-inference",   "relation": "共享帕累托优化的数学框架" },
    { "to": "rec-system",         "relation": "是推荐系统的子领域" }
  ],
  "papers": ["论文A", "论文B", "论文C", "论文F"]
}
```

**全部自然语言。** `description` 是路标，`relation` 是边的含义。LLM 读这些自然语言来决定导航方向。

**为什么是网不是树？**

| | 树 | 网 |
|---|---|---|
| 到达路径 | 唯一（从根到叶） | **多条**（从不同节点都能到达同一篇论文） |
| 发现关联 | 只能发现"同一分支下"的关联 | **能发现跨领域关联**（沿着边跳转） |
| 导航决策 | "进入哪个子节点" | **"走哪条边"** — 更丰富的推理空间 |
| 多归属 | 论文只能放一个位置 | **论文被多个节点引用**，天然解决 |
| 知识表达 | 层级分类（强制互斥） | **关系网络**（反映知识的真实结构） |

#### 第三层：LLM 导航 — 图上推理游走

推荐时，LLM 在语义网上做**推理驱动的游走**（不是树搜索，是图探索）：

```
导航过程（图上推理游走）

1. 读用户兴趣画像：
   "康熠最近在纠结多目标配平，辅助目标掉了。同时也在做冷启动项目。"

2. 读 Layer 0 顶层路标索引（~500 tokens）：
   [推荐系统] — 相关 → 进入
   [NLP] — 不相关 → 跳过
   [因果推断] — 暂不确定 → 记住，待会看有没有边连过来

3. 进入 [推荐系统]，读邻居节点 + 边描述（~800 tokens）：
   → [多目标优化]（子领域）— 直接相关 → 进入
   → [冷启动]（子领域）— 也相关 → 标记待访问
   → [用户建模]（子领域）— 不相关 → 跳过

4. 进入 [多目标优化]，读它的邻居 + 边（~600 tokens）：
   → 论文 A、B、C、F（包含）
   → [冷启动]（边："方法可用于冷启动场景"）
     ← LLM 推理："这条边说明有跨领域关联，值得跟"
   → [因果推断]（边："共享数学框架"）
     ← LLM 推理："和用户当前需求无关，跳过"

5. 沿 "方法可用于" 这条边到 [冷启动]，读摘要卡（~400 tokens）：
   发现论文 F 同时在这里
   → 推荐论文 F，理由："这篇同时解决多目标和冷启动，和你两个项目都相关"

6. 输出推荐列表 + 推荐理由

   总 token 消耗：500 + 800 + 600 + 400 = ~2300 tokens
   从 10 万篇中导航到最相关的几十篇
```

**每一步都是 LLM 推理**——读路标、读边的描述、理解语义关联、决定走哪条边。
系统只负责"打开 LLM 指定的那个节点"（数据导航），不做任何匹配判断。

**和树搜索的区别**：LLM 不是"从上往下走"，而是"从一个节点出发，沿着边探索"。可以跳转、回溯、发现意外关联。游走策略是推理，不是算法。

#### 为什么这不是外挂

关键判断标准：**谁在做决策？**

| | 标签交集 | 向量检索 | 语义路标 |
|---|---|---|---|
| 谁生成表示 | LLM | 外挂模型 | LLM |
| 谁做匹配决策 | **系统**（集合操作） | **系统**（余弦相似度） | **LLM**（推理导航） |
| 匹配逻辑在哪 | 硬编码在系统中 | 硬编码在系统中 | **在 LLM 的推理中** |

路标方案：**LLM 生成路标，LLM 消费路标。生成和消费都是基座能力。全链路零外挂。**

类比：图书馆书架标签写着"推荐系统 / 多目标优化"——这是组织方式（数据）。你看到标签后**自己决定**走到哪个书架——这是你的推理。没有系统在替你判断"你应该看这个书架"。

#### 与 embedding 方案的本质区别

| | embedding（外挂） | 标签交集（半外挂） | 语义路标（全基座） |
|---|---|---|---|
| 表示形式 | 黑盒向量 | 字符串标签 | **自然语言路标** |
| 生成者 | 外挂模型 | LLM | LLM |
| 消费者 | 系统（向量距离） | 系统（集合操作） | **LLM（推理导航）** |
| 模型升级 | 全量重 embed | 全量重生成标签 | 重生成路标即可 |
| 随基座加强 | ❌ | 生成加强，消费不变 | **生成 + 消费都加强** |
| 可解释性 | ❌ | 弱 | **✅ 路标和推荐理由都是自然语言** |

#### 复杂度对比

| 方案 | 论文入库成本 | 用户请求推荐成本 | 是否符合原则 |
|------|------------|----------------|-------------|
| 全量灌入（MVP） | O(0) | O(全量 tokens) | ✅ 但扛不住量 |
| fanout-on-write | O(用户数) | O(1) | ✅ 但扩散读 |
| ~~标签交集 + 精排~~ | O(1) | O(候选集) | ⚠️ 匹配是系统做的 |
| **语义路标 + LLM 导航** | **O(1)** | **O(log N × 路标)** | **✅ 全链路基座** |

10 万篇论文，3 级路标导航，LLM 只需 ~3500 tokens 即可到达目标。随上下文窗口增长，每级能看更多路标，导航更全面。

## 十一、路标系统设计（逐项讨论中）

### 关键认知：Inner Memory 就是小规模路标系统

OpenClaw 的 `workspace/memory/` 现在的结构：

```
Inner Memory 的路标结构（已验证可行）

memory/
├── MEMORY.md              ← 顶层路标（关键词 → 文件映射）
├── insights/
│   ├── creation-tools.md  ← 二级路标（洞察类）
│   └── ...
├── projects/
│   ├── openclaw-memory-reasoning.md  ← 二级路标（项目类）
│   └── ...
├── conversations.md       ← 时间线索引
└── ...
```

LLM（林风过竹）现在怎么用它的？

```
用户说"分身项目"
  → LLM 读 MEMORY.md（顶层路标）
  → 找到映射 → 加载 projects/openclaw-memory-reasoning.md
  → 在项目记忆里看到"关键洞察"引用
  → 跳转到 insights/creation-tools.md
  → 逐层深入，按需加载
```

**这就是路标导航。** LLM 读路标 → 推理决定走哪条路 → 读下一级。全程 LLM 推理，没有系统做匹配。而且这套系统现在 work 得很好。

所以：

> **Inner Memory 是路标系统在"小 size + 人工维护"条件下的特例。**
> **Outer Knowledge 的路标系统是同一套思路在"大 size + LLM 自动维护"条件下的泛化。**

| | Inner Memory（已有） | Outer Knowledge 路标（待建） |
|---|---|---|
| 规模 | 几十个文件 | 几万~几十万篇 |
| 维护者 | **人**（用户 + 分身协作整理） | **LLM**（入库时自动构建） |
| 路标形式 | MEMORY.md 映射 + 目录结构 | 多层语义网节点 + 边 |
| 导航者 | LLM | LLM |
| 数据格式 | 自然语言 Markdown | 自然语言路标 + 摘要卡 |

**导航者相同，导航方式相同。唯一的区别是谁来建和维护这些路标。**

**重要**：数据仍然分开。Inner Memory 的目标是构建"你自己"（身份、记忆、价值观），Outer Knowledge 的目标是组织"外部世界的信息"。**组织方式复用，数据独立。**

这意味着：
1. 不需要发明新的路标系统——泛化 Inner Memory 已有的模式即可
2. 数据格式可以复用（Markdown / JSON + 自然语言）
3. 核心要解决的是：**LLM 怎么自动维护路标**

### 已达成共识

以下设计决策已在讨论中确认：

**① 入库与结构变更解耦（原问题 1 + 2b + 4）**

```
日常入库（高频、轻量）              定期重整（低频、重量）
─────────────────────              ──────────────────────
新论文 → 摘要卡 → 归到现有节点      LLM 全局审视路标网络
                                     ↓
不改结构                           分裂 / 合并 / 新建边
不迁移论文                         迁移论文 + 更新路标描述
O(1) 推理                          O(N) 推理，但频率低（周/月）
```

- **入库时**：LLM 只做两件事——生成摘要卡 + 归到现有节点。即使归类不完美，暂挂最近似节点，等重整时再调
- **重整时**：LLM 全局审视——分裂过大节点、合并趋同节点、发现新的跨领域边、迁移受影响论文、更新过时的路标描述
- **路标过时问题**被重整机制自然吸收——重整时一并更新路标描述
- **物品迁移问题**被慢迁移解决——日常惰性不动，重整时批量归位
- **和人的类比**：每天随手归档到"大概是这个方向"的文件夹，定期花半天重新整理

**② 多层网状结构（原问题 3）**

- 路标系统不是树，是**多层语义网**
- **节点**：知识概念 + 自然语言路标描述
- **边**：节点间关系，自然语言描述（"方法可用于"、"共享数学框架"、"是子领域"）
- **论文多归属**：同一篇论文可被多个节点引用，天然解决归类冲突
- **导航方式**：LLM 在图上推理游走——纵向逐层深入（上层→下层），横向跨领域跳转（沿边游走）
- **层级是分裂的自然产物**：不预定义层数。热门方向论文多→分裂多→自然长得更深；冷门方向保持浅层。分裂机制保证每个叶子节点下的论文数在 LLM 可处理范围内

**层级导航的效率分析**：

```
假设每个节点最多挂 20 篇论文，每次分裂成 3-5 个子节点：

  Layer 0:  1 个根节点（索引）
  Layer 1:  ~5 个大方向           → 覆盖 100 篇
  Layer 2:  ~25 个细分方向         → 覆盖 500 篇
  Layer 3:  ~125 个更细节点        → 覆盖 2500 篇
  Layer 4:  ~625 个节点            → 覆盖 12500 篇

3-4 层就能覆盖上万篇论文。
每层 ~500-800 tokens（读子节点描述）+ ~50 tokens（输出选择）
4 层总计 ~3500 tokens

对比：直接读 2500 篇摘要卡 = ~250,000 tokens（不可能）
```

延迟评估：4 层 × 每层 1-2 秒 = 4-8 秒。日常 briefing 完全可接受。实时场景可优化：

| 优化方式 | 做法 | 性质 |
|----------|------|------|
| **合并层级读取** | 一次性读前 2 层，LLM 一步跳到 Layer 2，串行次数减半 | 用更大单次上下文换更少调用。上下文越大越 free |
| **缓存热门路径** | 系统记住常走路径，下次跳过前几层 | 系统能力（缓存），不是外挂判断 |
| **靠基座变强** | 上下文 128K→1M→10M，可一次性读更多层 | 长期自然解决 |

**③ Inner Memory 与 Outer Knowledge 的关系（原问题 5）**

- Inner Memory = 构建"你自己"，Outer Knowledge = 组织"外部世界的信息"
- **组织方式复用，数据独立**——不共享知识地图，避免污染身份边界
- Inner Memory 本身就是一个已验证的小规模路标系统，Outer Knowledge 是对同一模式的泛化
- 推荐时 LLM 同时读两者：Inner Memory 指导导航方向，Outer Knowledge 提供导航空间

```
推荐 = f(Inner Memory × Outer Knowledge)
       ↑ 你是谁         ↑ 世界发生了什么
```

### 开放问题

以下问题尚未解决，需要逐个讨论：

#### 开放问题 1：自生长的触发和节奏（已确认）

语义网的结构（节点数、层级、边）由 LLM 自生长，不预定义分类体系。

##### 已确认：两阶段触发

自生长不需要 LLM 一次性做"高难度自组织决策"。拆开看，每个子步骤都是 LLM 已经能做的事：

- **感知**（阅读理解 + 聚类）：读 20 篇摘要卡，判断"其中 8 篇讲多目标，7 篇讲冷启动" → 基座能力
- **判断**（决策）：看到节点有 30 篇且内部分成 3 个子群，判断"该拆了" → 基座能力
- **执行**（知识组织 + 命名）：给新节点写路标描述、给边写关系描述 → 基座能力

难点不在推理能力，在**触发时机**。用两阶段解决：

```
阶段 1：日常入库时的"微感知"（轻量，每次入库都做）
──────────────────────────────────────────────────────
LLM 归类论文到节点时，顺便输出 side signal：

  {
    "归类到": "推荐系统",
    "归类置信度": "低",              ← LLM 自己说"不太确定放哪"
    "感知信号": "这篇和节点下其他论文主题不太一样"
  }

这个信号不触发任何操作，只是"记一笔"。
成本：几乎为零——LLM 归类时多输出几个 token。
关键：置信度是 LLM 自己输出的判断，不是系统计算的 → 不违反设计原则。

阶段 2：重整时的"全局审视"（重量，低频触发）
──────────────────────────────────────────────
触发条件（满足任一即可）：
  a) 某节点积累了超过 N 篇论文（简单阈值，如 20 篇）
  b) 某节点近期"低置信度归类"比例超过 30%（LLM 不确定的太多了）
  c) 定时兜底（每月触发一次，不管有没有信号）

触发后 LLM 做什么：
  → 读该节点的所有摘要卡
  → 判断：要不要分裂？分成几个？怎么命名？
  → 输出结构变更 diff
```

**分工**：感知和判断由 LLM 做（基座能力），触发调度由系统做（系统能力）。符合设计原则。

##### 已确认：冷启动从零开始

```
第 0 天：语义网为空
  → 第 1 篇论文入库，LLM 创建第 1 个节点

前 N 篇（冷启动期）：
  → 节点很少（1-3 个），LLM 归类容易，置信度高
  → 不需要重整，结构简单

积累到阈值（~20 篇，约 1-2 周）：
  → 触发第一次重整
  → 结构自然"长出来"
```

不用种子节点。理由：
1. 冷启动期论文本来就少，归类粗糙的代价极低
2. 种子节点本质是"预定义"，和自生长精神矛盾
3. 第一次重整很快到来（1-2 周），届时结构自然分化

##### 完整的自生长生命周期

```
  第 0 天：冷启动
  ────────────────
  第 1 篇论文 → LLM 创建 [推荐系统] 节点 → 论文 A 挂上

  前 1-2 周（冷启动期）
  ────────────────────
  大部分论文挂在同一个节点下。归类简单，无需重整。

  积累到 ~20 篇
  ──────────────
  近期入库的论文归类置信度偏低（LLM 说"不太确定放哪"）
  系统检测到信号，触发第一次重整

  重整过程
  ────────
  LLM 读 [推荐系统] 下 20+ 篇摘要卡 →
  "这些论文在讲 3 件不同的事：
    a) 多目标优化（8 篇）
    b) 冷启动（7 篇）
    c) 其他推荐技术（5 篇）"

  → 分裂：
    [推荐系统] 保留 5 篇（通用的）
    [推荐系统] --"子领域"--> [多目标优化] 8 篇
    [推荐系统] --"子领域"--> [冷启动] 7 篇

  → LLM 还发现：
    "多目标优化里有 2 篇和冷启动有交集"
    → 建边 [多目标优化] ══"方法可用于"══ [冷启动]
    → 这 2 篇论文同时挂在两个节点下

  分裂后的日常
  ──────────────
  新论文入库，LLM 读路标索引（3 个节点描述）→ 归到具体节点
  归类更精准（选项从 1 个变成 3 个），置信度回升

  继续积累 → 下一次重整 → 循环
```

##### 基座能力评估

每个子步骤现在的模型都能做：读摘要卡做聚类 ✅、给聚类命名 ✅、判断是否需要拆分 ✅、写路标描述 ✅。

全流程自动化需要的是 **prompt 工程 + 结构化输出**（JSON schema）——这属于系统能力范畴。LLM 做判断，系统做编排。

**✅ 状态：已确认——两阶段触发 + 从零冷启动 + LLM 判断/系统调度分工**

#### 开放问题 2：边的设计（已确认）

##### 边的类型：LLM 自由创建，不预定义

信任基座升级，边的关系描述完全由 LLM 自由生成，不预定义类型（如"子领域"、"方法可用于"）。

理由：
- 预定义边类型 = 硬编码知识结构 = 和预定义分类体系同样的问题
- LLM 自由创建和节点路标描述的设计一致——都是自然语言，都由 LLM 生成和消费
- 基座越强，边的描述越精准（能发现更微妙的关系）
- 描述质量不稳定的风险（同一种关系被描述成不同的话）由重整时统一修正

##### 稠密度控制：不需要单独机制，节点分裂天然分散边

问题 1 的分裂机制已间接解决了稠密度问题：

```
为什么一个节点会有 15 条边？
  → 因为它涵盖的范围太广，和很多细分方向都有关系

当它分裂成子节点后：
  → 原节点变成更抽象的上层节点，边大幅减少
  → 边被"分散"到子节点上
  → 每个子节点的邻居数自然合理

             分裂前                        分裂后

  [A]─[B]─[C]─[D]─[E]              [A]─[B]
       │    │    │                       │
       └─[推荐系统]─┘              [推荐系统]（上层，只留抽象边）
         │  │  │                    ├─[多目标]─[C]─[D]
       [F]─[G]─[H]                 └─[冷启动]─[E]
                                        │
       15 条边 → token 膨胀              [F]─[G]─[H]
                                   
                                   每个节点 3-5 条边 → 合理
```

极端情况（如"Transformer"这种真正的枢纽概念）：分裂也没用，它就是和很多方向有关。但 10-15 条边描述，每条几十 tokens，总共几百 tokens——在当前上下文窗口下可承受，且模型越来越强、上下文越来越大，上限自然放宽。

**结论：节点分裂天然分散边，极端枢纽靠基座上下文能力承受。不需要额外机制。**

##### 边的生命周期：跟着重整走

边不需要独立的生命周期管理，跟着重整统一处理：

```
重整时 LLM 对边做什么：

1. 检查边描述是否还准确 → 重写过时的描述
2. 检查边是否还有存在的必要 → 删除失效的边
3. 发现新的关联 → 创建新边
4. 节点合并/分裂时的边迁移：
   - 合并 → 两组边合并或重写
   - 分裂 → 边被分配到子节点上
```

边的增删改都发生在重整时——和节点分裂/合并、论文迁移是同一个过程。符合慢迁移共识，减少系统复杂度。

**✅ 状态：已确认——LLM 自由创建边、分裂天然控制稠密度、生命周期跟着重整走**

#### 开放问题 3：重整的具体机制（已确认）

已在问题 1、2 的讨论中回答了大部分子问题：

| 子问题 | 回答 | 出处 |
|--------|------|------|
| **重整的输入** | 读触发节点下的所有摘要卡 | 问题 1：阶段 2 全局审视 |
| **重整的输出** | 结构变更 diff（新增/删除节点、边变更、论文迁移） | 问题 1：阶段 2 |
| **重整的范围** | 默认增量（针对触发的节点），定时兜底时全局 | 问题 1：触发条件 a/b 是局部，c 是全局 |
| **重整的质量保证** | 暂不需要额外机制 | 见下 |

**质量保证为什么暂时不需要**：
1. 短期数据量小（几十到几百篇），重整结果一眼可判
2. 用户就是自己，推荐准不准直接体感反馈
3. 重整是 LLM 推理，质量随基座加强自然提升
4. 如果需要验证，最自然的方式是重整后让 LLM 自己 review——也是基座能力

**✅ 状态：已确认——前三个子问题已在问题 1/2 中回答，质量保证暂不需要额外机制**
