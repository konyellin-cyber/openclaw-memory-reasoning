# 个人分身推荐系统 — 方案讨论

> 用 100% 的你去匹配整个世界

**状态**: v0.1 已验证，核心课题讨论中  
**日期**: 2026-02-28  
**参与**: 作者 + AI 分身

---

## 一、项目是什么

一个运行在 OpenClaw 上的个人信息推荐系统。

分身（林风过竹）不只记得你说过什么（Inner Memory），还能看到外面发生了什么（Outer Knowledge），然后主动把对你有价值的信息推荐给你。

**核心公式**：

```
推荐 = f(你是谁, 你在想什么, 外面发生了什么)
     = f(Inner Memory × Outer Knowledge)
```

---

## 二、为什么要做这个

### 思路演变

| 阶段 | 方向 | 问题 |
|------|------|------|
| 2/21 | 双塔向量检索（外挂记忆召回系统） | 补偿层 — 模型长上下文能力变强后被淘汰 |
| 2/28 上午 | 轻量模型做记忆路由推理 | 也是补偿层 — 模型变强后自己能路由 |
| **2/28 下午** | **个人分身推荐系统** | **模型越强推荐越准 — 壁垒在记忆积累和信息源品味** |

### 核心判断

向量检索、记忆路由都是"补偿模型暂时做不好的事"。
个人信息推荐是"模型再强也替代不了的系统能力"——
因为模型没有持久状态、没有外部信息源、不会主动找你。

### 为什么现在做

- 自己就是用户，闭环验证，反馈环路极短
- Inner Memory 已经积累了（OpenClaw workspace）
- 推荐系统工作经验可直接迁移
- 完全个人项目，零公司依赖，不会有"精力分散"观感
- 周末就能跑通最小切片

## 三、与传统推荐系统的同构映射

这个项目和推荐系统领域的工作高度同构，但服务对象完全不同：

| 概念 | 公众号推荐（工作） | 个人分身推荐（本项目） |
|------|-------------------|---------------------|
| 用户画像 | 行为特征 + 标签 | **完整个人记忆 + 价值观** |
| 物品池 | 公众号文章 | arXiv 论文 + RSS 内容 |
| 召回 | 多路召回 → 粗筛 | Inner/Outer 双源融合 |
| 排序 | CTR/CVR 预估 | **"对你的价值"预估** |
| 多目标 | 点击 × 时长 × 互动 | 相关性 × 新颖性 × 时效性 × 行动力 |
| 探索/利用 | 兴趣探索 vs 确定性推荐 | "你不知道但应该知道" vs "你正在关注的" |
| 冷启动 | 新用户 / 新内容 | 新信息源 / 新兴趣领域 |

传统推荐系统是"用 1% 的你去匹配 100% 的内容"。
这个系统是"用 100% 的你去匹配整个世界"。

## 四、双源信息模型

### Inner Memory（已有）

来源：OpenClaw `workspace/memory/`

包含：
- 个人记忆、价值观、偏好
- 决策历史、思维方式
- 项目上下文、当前关注点
- 洞察与方法论

不需要额外建设，直接复用。

### Outer Knowledge（新增）

#### Phase 1 起步：arXiv cs.IR

- **RSS Feed**: `http://arxiv.org/rss/cs.IR`（Information Retrieval）
- **频率**: 每天 5-20 篇新论文
- **数据**: 标题 + 作者 + 摘要（不需要全文）
- **合法性**: ✅ 完全公开，arXiv 官方原生 RSS
- **为什么选这个**: 和专业高度相关，信息密度高，结构化好

#### Phase 2 扩展（后续按需加）

| 类别 | 来源 | 获取方式 |
|------|------|---------|
| 学术 | arXiv cs.AI / cs.LG | 原生 RSS |
| 技术博客 | 美团/字节/阿里技术博客 | 原生 RSS |
| 中文社区 | 知乎专栏、即刻圈子 | RSSHub |
| 国际社区 | Hacker News、Reddit ML | 原生 RSS |
| 行业媒体 | 机器之心、量子位 | RSSHub |

**原则**: 管道通用（标准 RSS 协议）、源可插拔、零公司依赖。

## 五、产品形态

直接在 OpenClaw 上实现，不做额外 app/bot。

### 对话中自然推荐

聊天过程中，agent **自己判断**当前话题和 feed 中的论文相关，**主动调用工具查找**，在回复中自然带出推荐：

```
你：最近多目标优化搞得头疼，一个实验的辅助目标指标掉了不少...

AI 分身：确实，不同优化目标之间天然有冲突...

  对了，最近有两篇论文和你正在纠结的事情直接相关：
  1. 📄 "Pareto-Optimal Multi-Objective Reward Shaping for Recommendation"
     → 这篇提出了一种不需要手动调 loss 权重的多目标配平方法，
       和常见的多任务塔耦合思路不同，可能有启发
  2. 📄 "Constrained Optimization for Ecosystem Diversity in RecSys"
     → 直接讲模态挤占问题的，和"主模态涨、副模态跌"的现象完全对应
```

**没有任何外挂判断层**。Agent 自己理解上下文，自己决定要不要调 `search_feed` 工具，自己把结果融进回复。

**技术原理**：

```
用户消息 → agent 正常推理
              ↓
        agent 自主判断：当前话题和 feed 内容可能相关
              ↓
        agent 调用 tool: search_feed({ days: 7 })
              ↓
        tool 返回匹配的论文（标题 + 摘要）
              ↓
        agent 在回复中自然带出推荐 + 推荐理由
```

**为什么这样最好**：
- **零额外组件** — 不需要 hook、不需要关键词表、不需要 embedding 索引、不需要 `chat.inject`
- **模型越强越好** — 判断越准、推荐理由越深、融入对话越自然
- **最自然的交互** — 推荐就在回复里，不是"异步注入"的外来物
- **一切能力基于 LLM 基座进化** — 和项目核心判断完全一致

### 反馈闭环

- 你点开/忽略的论文 → 记录偏好
- 你的评价（"这篇有用" / "不相关"）→ 调整推荐
- 随时间积累，推荐越来越准

## 六、系统架构

```
┌───────────────────────────────────────────────────────────┐
│              Personal Rec-System (OpenClaw Plugin)          │
│                                                             │
│  Inner Memory（已有）          Outer Knowledge（新增）       │
│  └── workspace/memory/*        └── RSS Feed 后台拉取        │
│         │                             │                     │
│         ▼                             ▼                     │
│  ┌─────────────────────────────────────────────────┐       │
│  │        LLM 基座（agent 自主推理）                  │       │
│  │                                                   │       │
│  │  agent 在正常对话中自主判断：                       │       │
│  │  "当前话题是否和 feed 中的内容相关？"               │       │
│  │  → 是：调用 search_feed tool → 拿到匹配论文       │       │
│  │  → 在回复中自然带出推荐 + 推荐理由                 │       │
│  │                                                   │       │
│  │  没有外挂判断层，一切交给 LLM 基座。               │       │
│  └──────────────┬──────────────────────────────────┘       │
│                 ▼                                           │
│  ┌─────────────────────────────────────────────────┐       │
│  │        Plugin 提供的能力                          │       │
│  │                                                   │       │
│  │  registerService → 后台定时拉取 RSS               │       │
│  │  registerTool: search_feed → 检索本地 feed 数据   │       │
│  │                                                   │       │
│  │  反馈: memory → 记录偏好，持续优化                  │       │
│  └─────────────────────────────────────────────────┘       │
└───────────────────────────────────────────────────────────┘
```

核心就三件事：
1. **拉数据** — `registerService` 后台从 RSS 拉公开内容，存到本地
2. **提供工具** — `registerTool` 注册 `search_feed`，让 agent 能检索 feed 数据
3. **agent 自主推荐** — agent 在对话中自己决定调不调工具、怎么融入回复

## 七、设计原则（铁律）

| 原则 | 说明 |
|------|------|
| **一切基于 LLM 基座** | 不做外挂检索/向量召回/路由层/触发判断层，agent 自主推理和调用工具 |
| **不做补偿层** | 如果一个能力属于"模型能力的子集"，模型变强后就会淘汰它——不投资 |
| **Plugin 只提供系统能力** | 拉数据、存数据、注册工具。推理和判断全部交给 LLM 基座 |
| **个人项目定位** | 不依赖任何私有数据或内部接口 |
| **公开数据源** | 所有数据通过公开合法渠道获取（RSS） |
| **产品形态用 OpenClaw** | 不做额外 app/bot/CLI |
| **方便分发** | npm 安装即可用，任何人都能试用 |
| **渐进式** | 先跑通一个信息源（arXiv），再扩展 |

**判断标准**：设计任何新功能前先问——"这件事模型足够强后能自己做吗？"
- 能 → 不做（补偿层）
- 不能 → 做（系统能力：持久化、定时拉取、外部数据源、反馈记录）

## 八、实现决策

### 1. 实现形态：Plugin（已确定）

**核心需求**：代码级能力 + 任何人（公司内外）都能方便安装试用。

**排除的方案**：
- ~~Skill~~：声明式（Markdown SOP），无法注册 cron/hooks，能力不够
- ~~独立脚本~~：不走 OpenClaw 扩展机制，用户需要手动配 cron，安装体验差

**选择 Plugin 的理由**：

| 能力 | Plugin 支持 |
|------|------------|
| 注册定时任务（cron） | ✅ `api.registerService()` / hooks |
| 注册 AI 工具 | ✅ `api.registerTool()` |
| 注册生命周期钩子 | ✅ `api.on("session_start", ...)` 等 20+ 事件 |
| 读写 workspace 文件 | ✅ |
| 调用 LLM | ✅ `runEmbeddedPiAgent()` |
| 自定义配置 | ✅ `openclaw.plugin.json` 的 `configSchema` |

**分发方式**：

```bash
# 1. npm 全局安装
npm install -g openclaw-plugin-personal-rec

# 2. 在 openclaw.json 中启用
{
  "plugins": {
    "entries": {
      "personal-rec": {
        "enabled": true,
        "config": {
          "feeds": ["http://arxiv.org/rss/cs.IR"]
        }
      }
    }
  }
}
```

也支持手动安装（放到 `~/.openclaw/extensions/personal-rec/`），以及通过配置 `plugins.load.paths` 指定本地开发目录。

### 2. 触发判断（已确定：交给 LLM 基座）

**不做任何外挂判断层**。

~~关键词匹配~~、~~轻量 LLM 分类~~、~~Embedding 相似度~~、~~混合策略~~ — 全部排除。

Agent 在正常对话推理过程中自己决定要不要调 `search_feed` 工具。
这就是 LLM 基座做意图理解 — 模型越强，判断越准。

Plugin 只需要把工具注册好，附上清晰的 tool description，剩下的交给 agent。

### 3. 存储格式

```
~/.openclaw/personal-rec/feeds/
├── 2026-02-26.json    # 每日一个文件
├── 2026-02-27.json
└── ...
index.json             # 全局索引（日期范围、论文数等元信息）
```

每篇论文的数据结构：

```json
{
  "id": "2602.12345",
  "title": "...",
  "authors": ["..."],
  "abstract": "...",
  "date": "2026-02-28",
  "source": "arxiv:cs.IR",
  "url": "https://arxiv.org/abs/2602.12345"
}
```

JSON 而不是 Markdown — 因为后续演进（摘要卡、标签索引）需要结构化数据。
存在 Plugin 自己的 data 目录而不是 workspace — 因为这是"物料"不是"记忆"。

### 4. 推荐效果怎么衡量？

自己是唯一用户，没有统计学意义的 A/B 实验。
怎么判断推荐越来越准了？主观感受？还是设计一套简单的反馈记录？
