## 一、项目是什么

一个运行在 OpenClaw 上的个人信息推荐系统。

分身（林风过竹）不只记得你说过什么（Inner Memory），还能看到外面发生了什么（Outer Knowledge），然后主动把对你有价值的信息推荐给你。

**核心公式**：

```
推荐 = f(你是谁, 你在想什么, 外面发生了什么)
     = f(Inner Memory × Outer Knowledge)
```

## 二、为什么要做这个

### 思路演变

| 阶段 | 方向 | 问题 |
|------|------|------|
| 2/21 | 双塔向量检索（外挂记忆召回系统） | 补偿层 — 模型长上下文能力变强后被淘汰 |
| 2/28 上午 | 轻量模型做记忆路由推理 | 也是补偿层 — 模型变强后自己能路由 |
| **2/28 下午** | **个人分身推荐系统** | **模型越强推荐越准 — 壁垒在记忆积累和信息源品味** |

### 核心判断

向量检索、记忆路由都是"补偿模型暂时做不好的事"。
个人信息推荐是"模型再强也替代不了的系统能力"——
因为模型没有持久状态、没有外部信息源、不会主动找你。

### 为什么现在做

- 自己就是用户，闭环验证，反馈环路极短
- Inner Memory 已经积累了（OpenClaw workspace）
- 推荐系统工作经验可直接迁移
- 完全个人项目，零公司依赖，不会有"精力分散"观感
- 周末就能跑通最小切片

## 三、与传统推荐系统的同构映射

这个项目和工作在做的事高度同构，但服务对象完全不同：

| 概念 | 公众号推荐（工作） | 个人分身推荐（本项目） |
|------|-------------------|---------------------|
| 用户画像 | 行为特征 + 标签 | **完整个人记忆 + 价值观** |
| 物品池 | 公众号文章 | arXiv 论文 + RSS 内容 |
| 召回 | 多路召回 → 粗筛 | Inner/Outer 双源融合 |
| 排序 | CTR/CVR 预估 | **"对你的价值"预估** |
| 多目标 | 点击 × 时长 × 互动 | 相关性 × 新颖性 × 时效性 × 行动力 |
| 探索/利用 | 兴趣探索 vs 确定性推荐 | "你不知道但应该知道" vs "你正在关注的" |
| 冷启动 | 新用户 / 新内容 | 新信息源 / 新兴趣领域 |

传统推荐系统是"用 1% 的你去匹配 100% 的内容"。
这个系统是"用 100% 的你去匹配整个世界"。

## 四、双源信息模型

### Inner Memory（已有）

来源：OpenClaw `workspace/memory/`

包含：
- 个人记忆、价值观、偏好
- 决策历史、思维方式
- 项目上下文、当前关注点
- 洞察与方法论

不需要额外建设，直接复用。

### Outer Knowledge（新增）

#### Phase 1 起步：arXiv cs.IR

- **RSS Feed**: `http://arxiv.org/rss/cs.IR`（Information Retrieval）
- **频率**: 每天 5-20 篇新论文
- **数据**: 标题 + 作者 + 摘要（不需要全文）
- **合法性**: ✅ 完全公开，arXiv 官方原生 RSS
- **为什么选这个**: 和专业高度相关，信息密度高，结构化好

#### Phase 2 扩展（后续按需加）

| 类别 | 来源 | 获取方式 |
|------|------|---------|
| 学术 | arXiv cs.AI / cs.LG | 原生 RSS |
| 技术博客 | 美团/字节/阿里技术博客 | 原生 RSS |
| 中文社区 | 知乎专栏、即刻圈子 | RSSHub |
| 国际社区 | Hacker News、Reddit ML | 原生 RSS |
| 行业媒体 | 机器之心、量子位 | RSSHub |

**原则**: 管道通用（标准 RSS 协议）、源可插拔、零公司依赖。

## 五、产品形态

直接在 OpenClaw 上实现，不做额外 app/bot。

### 对话中自然推荐

聊天过程中，agent **自己判断**当前话题和 feed 中的论文相关，**主动调用工具查找**，在回复中自然带出推荐：

```
你：最近多目标配平搞得头疼，liujunwang 的加热实验掉了 18%...

林风过竹：确实，加热保量和 CTR 优化天然冲突。你们现在混排侧的摇臂参数...

  对了，最近有两篇论文和你正在纠结的事情直接相关：
  1. 📄 "Pareto-Optimal Multi-Objective Reward Shaping for Recommendation"
     → 这篇提出了一种不需要手动调 loss 权重的多目标配平方法，
       和你们现在 ESMM 塔耦合的思路不同，可能有启发
  2. 📄 "Constrained Optimization for Ecosystem Diversity in RecSys"
     → 直接讲模态挤占问题的，和 lemnali 文章涨→图片跌的现象完全对应
```

**没有任何外挂判断层**。Agent 自己理解上下文，自己决定要不要调 `search_feed` 工具，自己把结果融进回复。

**技术原理**：

```
用户消息 → agent 正常推理
              ↓
        agent 自主判断：当前话题和 feed 内容可能相关
              ↓
        agent 调用 tool: search_feed({ topic: "multi-objective optimization" })
              ↓
        tool 返回匹配的论文（标题 + 摘要）
              ↓
        agent 在回复中自然带出推荐 + 推荐理由
```

**为什么这样最好**：
- **零额外组件** — 不需要 hook、不需要关键词表、不需要 embedding 索引、不需要 `chat.inject`
- **模型越强越好** — 判断越准、推荐理由越深、融入对话越自然
- **最自然的交互** — 推荐就在回复里，不是"异步注入"的外来物
- **一切能力基于 LLM 基座进化** — 和项目核心判断完全一致

### 反馈闭环

- 你点开/忽略的论文 → 记录偏好
- 你的评价（"这篇有用" / "不相关"）→ 调整推荐
- 随时间积累，推荐越来越准

```
┌───────────────────────────────────────────────────────────┐
│              Personal Rec-System (OpenClaw Plugin)          │
│                                                             │
│  Inner Memory（已有）          Outer Knowledge（新增）       │
│  └── workspace/memory/*        └── RSS Feed 后台拉取        │
│         │                             │                     │
│         ▼                             ▼                     │
│  ┌─────────────────────────────────────────────────┐       │
│  │        LLM 基座（agent 自主推理）                  │       │
│  │                                                   │       │
│  │  agent 在正常对话中自主判断：                       │       │
│  │  "当前话题是否和 feed 中的内容相关？"               │       │

│  │                                                   │       │
│  │  没有外挂判断层，一切交给 LLM 基座。               │       │
│  └──────────────┬──────────────────────────────────┘       │
│                 ▼                                           │
│  ┌─────────────────────────────────────────────────┐       │
│  │        Plugin 提供的能力                          │       │
│  │                                                   │       │
│  │  registerService → 后台定时拉取 RSS               │       │
│  │  registerTool: search_feed → 检索本地 feed 数据   │       │
│  │                                                   │       │
│  │  反馈: memory → 记录偏好，持续优化                  │       │

└───────────────────────────────────────────────────────────┘
```

核心就三件事：
1. **拉数据** — `registerService` 后台从 RSS 拉公开内容，存到本地
2. **提供工具** — `registerTool` 注册 `search_feed`，让 agent 能检索 feed 数据
3. **agent 自主推荐** — agent 在对话中自己决定调不调工具、怎么融入回复

## 八、待讨论的问题

### 1. 实现形态：Plugin（已确定）

**核心需求**：代码级能力 + 任何人（公司内外）都能方便安装试用。

**排除的方案**：
- ~~Skill~~：声明式（Markdown SOP），无法注册 cron/hooks，能力不够
- ~~独立脚本~~：不走 OpenClaw 扩展机制，用户需要手动配 cron，安装体验差

**选择 Plugin 的理由**：

| 能力 | Plugin 支持 |
|------|------------|
| 注册定时任务（cron） | ✅ `api.registerService()` / hooks |
| 注册 AI 工具 | ✅ `api.registerTool()` |
| 注册生命周期钩子 | ✅ `api.on("session_start", ...)` 等 20+ 事件 |
| 读写 workspace 文件 | ✅ |
| 调用 LLM | ✅ `runEmbeddedPiAgent()` |
| 自定义配置 | ✅ `openclaw.plugin.json` 的 `configSchema` |

**分发方式**：

Plugin 支持 npm 包分发，用户安装只需两步：

```bash
# 1. npm 全局安装
npm install -g openclaw-plugin-personal-rec

# 2. 在 openclaw.json 中启用
{
  "plugins": {
    "entries": {
      "personal-rec": {
        "enabled": true,
        "config": {
          "feeds": ["http://arxiv.org/rss/cs.IR"]
        }
      }
    }
  }
}
```

也支持手动安装（放到 `~/.openclaw/extensions/personal-rec/`），以及通过配置 `plugins.load.paths` 指定本地开发目录。

### 2. `search_feed` 工具设计 — 核心课题

这个工具是整个项目最有意思的部分：**怎么用 LLM 推理在海量数据中做检索和推荐？**

传统路径是 embedding → 向量检索 → top-K → 排序模型。
这个项目要探索的是：**纯 LLM 推理能不能替代这条链路？**

#### MVP：全量灌入，agent 自己筛

```typescript
registerTool({
  name: "search_feed",
  description: "搜索最近拉取的学术论文。当你觉得当前对话和学术研究可能有关时调用。",
  parameters: {
    days: { type: "number", description: "最近几天，默认 7" }
  },
  handler: async ({ days = 7 }) => {
    // 读本地 feed 文件 → 按时间过滤 → 全量返回
    return papers; // [{title, authors, abstract, date, url}]
  }
});
```

数据量：arXiv cs.IR 每天 5-20 篇，一周 ~100 篇，标题+摘要 ~15K tokens。
当前模型的上下文窗口完全能吃下。agent 拿到全量数据后自己判断哪些和当前对话相关。

**这个阶段验证的是**：LLM 能不能在 ~100 篇论文中准确找到和当前话题相关的 2-3 篇？

#### 演进方向：数据量增长后怎么办？

当 feed 源扩展到多个（arXiv + 技术博客 + 社区），积累几个月，数据量可能到 **数千甚至上万篇**。
这时候全量灌给 agent 不现实了。怎么办？

这就是核心研究课题——**用 LLM 推理做检索**，而不是回退到传统向量检索。几个可能的方向：

**方向 A：分层推理**
```
全量 feed → LLM 第一轮：按话题/领域做粗分类（可离线/预计算）
              ↓
agent 调 search_feed 时 → 只拉相关分类的子集 → 全量灌入
```
类似推荐系统的召回→精排，但每一层都是 LLM 推理。

**方向 B：摘要压缩**
```
每日 feed → LLM 生成当日摘要（"今天 cs.IR 有 15 篇，主要方向是..."）
              ↓
agent 先看摘要 → 判断要不要深入 → 再调工具拉具体论文
```
两次 tool call，先粗后细，模拟人类"先扫标题再细看"的行为。

**方向 C：LLM 自主分页**
```
search_feed 返回第一页（20篇）+ 总数
agent 看完判断 → 要不要翻下一页？
                → 要不要换关键词重查？
```
让 agent 像人一样翻页浏览，自己控制检索深度。

**方向 D：记忆化索引**
```
每篇论文入库时 → LLM 给它打标签（和 Inner Memory 中哪些关注点相关）
              ↓
search_feed 时 → 按标签过滤 → 返回子集
```
标签是 LLM 生成的，不是向量，可读可调试。

**这些方向不是现在要选的，而是做完 MVP 后自然会遇到的问题。**
有意思的地方在于：每个方向都是"用 LLM 推理替代传统检索的某个环节"，而不是回退到 embedding/向量。

#### 存储格式（为演进预留）

MVP 用最简单的方式：

```
~/.openclaw/extensions/personal-rec/data/
├── feeds/
│   ├── 2026-02-28.json    # 每日一个文件
│   ├── 2026-03-01.json
│   └── ...
└── index.json             # 全局索引（日期范围、论文数等元信息）
```

每篇论文的数据结构：

```json
{
  "id": "2602.12345",
  "title": "...",
  "authors": ["..."],
  "abstract": "...",
  "date": "2026-02-28",
  "source": "arxiv:cs.IR",
  "url": "https://arxiv.org/abs/2602.12345"
}
```

JSON 而不是 Markdown — 因为后续演进（分层推理、标签索引）需要结构化数据。
存在 Plugin 自己的 data 目录而不是 workspace — 因为这是"物料"不是"记忆"。

### 4. 触发判断（已确定：交给 LLM 基座）

**不做任何外挂判断层**。

~~关键词匹配~~、~~轻量 LLM 分类~~、~~Embedding 相似度~~、~~混合策略~~ — 全部排除。

Agent 在正常对话推理过程中自己决定要不要调 `search_feed` 工具。
这就是 LLM 基座做意图理解 — 模型越强，判断越准。

Plugin 只需要把工具注册好，附上清晰的 tool description，剩下的交给 agent。

### 5. 推荐效果怎么衡量？

自己是唯一用户，没有统计学意义的 A/B 实验。
怎么判断推荐越来越准了？主观感受？还是设计一套简单的反馈记录？

## 九、最小可运行切片（MVP）

**目标**：聊天中聊到相关话题时，agent 自己调工具查 feed，在回复中自然带出论文推荐。

### 需要做的事（按顺序）：

1. **RSS 拉取**：`registerService` 后台拉取 `http://arxiv.org/rss/cs.IR`，提取论文标题 + 摘要
2. **本地存储**：Feed 数据存到本地（格式待定）
3. **注册工具**：`registerTool` 注册 `search_feed`，让 agent 能检索 feed
4. **跑通对话**：和 agent 聊推荐系统相关话题 → agent 自主调用 `search_feed` → 在回复中带出推荐

### 不需要做的事（MVP 排除）：

- 多信息源支持
- 反馈记录
- 历史数据管理
- 推荐效果评估

先跑通"聊天 → agent 自主调工具 → 推荐融入回复"这条最短路径。
